# AutoTest Hub

AutoTest Hub — Windows 开箱即用的单人专属自动化测试工具。

## 技术栈

- **后端**：Python 3.11+ / FastAPI / SQLAlchemy / SQLite
- **前端**：Vue 3 / Element Plus / Electron
- **自动化**：Playwright / Appium / PaddleOCR

## 代码规范

### 文件结构
- 文件命名：全小写 + 下划线
- Python 文件头部：`# -*- coding: utf-8 -*-`
- 工具函数顶部写 `# @Function:` 注释

### 命名规范
- 类名：`MyClass`（大驼峰）
- 函数名：`my_function`（小写下划线）
- 常量：`MY_CONSTANT`（全大写）

### 函数规范
- 必须有 docstring
- 参数和返回值要有类型提示
- 职责单一，不超过 50 行

### 格式规范
- 缩进：4 个空格
- 行宽：最大 120 字符
- 字符串：双引号优先
- 尾部逗号：多行数据结构最后一项加逗号

## Git 规范

### Commit Message
```
<type>(<scope>): <subject>
```

| 类型 | 说明 | 示例 |
|------|------|------|
| feat | 新功能 | `feat(auth): 添加登录功能` |
| fix | Bug 修复 | `fix(api): 修复超时问题` |
| docs | 文档更新 | `docs: 更新 README` |
| style | 代码格式 | `style: 统一缩进` |
| refactor | 重构 | `refactor(core): 提取基类` |
| test | 测试相关 | `test: 添加登录测试` |
| chore | 构建/工具 | `chore: 更新依赖` |

### 分支命名
```
feature/用户登录
fix/超时问题
docs/更新README
```

### 提交前检查
- 代码能正常运行
- 没有硬编码敏感信息
- 文档已同步更新

## 安全规范

### 绝对禁止
- 不得提交 API Key、密码、密钥
- 不得提交 .env 文件
- 不得在代码中硬编码任何凭证

### 发现安全问题时
1. 立即停止提交
2. 告知用户
3. 清理 Git 历史（如已提交）

## 对话管理规则

### 重要：每次对话都是独立的

**⚠️ 关键规则：**
- 每次新对话开始时，**不要假设之前对话的状态**
- **不要继续执行之前对话中未完成的任务**
- **不要引用之前对话中的决策或上下文**
- **每次对话都需要用户明确告知要做什么**

### 对话开始时
1. 等待用户明确说明本次对话的目标
2. 如果用户没有说明，主动询问："请问这次对话需要我帮你做什么？"
3. 不要自动执行任何操作

### 对话结束时
1. 确认用户是否还有其他需求
2. 不要自动提交代码或执行其他操作

## 自动 Skill 形成规则

### 触发条件
当用户多次执行相同类型的任务时，AI 应该：
1. 识别重复的操作模式
2. 建议创建新的 Skill
3. 与用户确认后，自动创建 Skill 文件

### 自动形成流程

#### 1. 识别模式
当发现以下情况时，考虑创建 Skill：
- 用户多次执行相同的操作（如"创建数据库表"、"添加 API 接口"）
- 某类任务需要多个步骤才能完成
- 用户经常询问相同类型的问题

#### 2. 建议创建
向用户说明：
- 发现了重复的操作模式
- 建议创建 Skill 来简化操作
- 询问用户是否同意

#### 3. 创建 Skill
如果用户同意，在 `.claude/skills/` 目录创建新的 Skill 文件：

```markdown
# Skill 名称

> 简短描述

## 触发条件
- 用户说"xxx"时
- 用户做"xxx"时

## 执行步骤
1. 第一步
2. 第二步
3. ...

## 示例
用户：xxx
AI：xxx
```

#### 4. 更新配置
创建 Skill 后，需要：
1. 更新 `.claude/skills.json`（如果存在）
2. 告知用户新的 Skill 已创建
3. 说明如何使用

### Skill 命名规范
- 使用小写字母和连字符
- 名称应该简洁明了
- 例如：`create-api`、`add-database`、`deploy-service`
